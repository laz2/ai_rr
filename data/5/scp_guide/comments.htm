Оформление комментариев

 

Желательно комментировать каждый оператор (пусть даже не подробно), так как в SCP почти каждый оператор выполняет сложные составные действия и иногда смысл их действий нетривиален. Поэтому для облегчения понимания смысла программы другими программистами и для глубокого понимания собственных разрабатываемых программ желательно не игнорировать данный на первый взгляд не обязательный шаг. Поэтому вопросу оформления комментариев выделена отдельная глава.

Важность написания комментариев понятна из следующего факта: написание комментариев принуждает к описанию программного кода на естественном языке, это провоцирует воображение и можно, объясняя самому себе ход программы, предварительно увидеть будущие «тупики» и «подводные камни» в алгоритме. Кроме того, при описании программного кода чаще возникают творческие мысли по усовершенствованию алгоритма и расширению функционала разрабатываемой программы.

 

Правила оформления комментариев необходимы для универсализации написания программного кода и как следствие повышения его удобочитаемости и понимания:

 

<заголовок программы>

<пустая строка>

///////////////////////////////////////////////////////////////////////////////

// <Первая строка комментария>

<оператор>

///////////////////////////////////////////////////////////////////////////////

// <Первая строка комментария>

// <Вторая строка комментария>

// <и т.д.>

<оператор>

…

 

После строк <оператор> можно ставить пустые строки – это улучшит разделение операторов. Но более удобным методом представляется следующий: не ставить пустые строки между связанными по смыслу и назначению операторами, то есть следует выделить в программе блоки операторов, выполняющих определённые подзадачи. Тогда блоки следует разделять пустыми строками, а внутри блоков будет находиться сплошной текст из чередующихся операторов и комментариев, описывающих эти операторы. Тогда в программе будут чётко видны смысловые блоки, и навигация упростится, и понимание структуры программы улучшится. Причём важные и крупные блоки можно комментировать следующим способом:

 

…

<предыдущий блок операторов>

<пустая строка>

////////////////////////////////////////////////////////////////////////////////

//                 <Отцентрированный текст комментария>

////////////////////////////////////////////////////////////////////////////////

<пустая строка>

<блок операторов, который описывает комментарий выше>

…

 

Обратите внимание на строку со слешами. Её использование не случайно – эта строка хорошо выделяется среди остальных и помогает легко ориентироваться в программном коде, так как операторы и соответствующие им комментарии хорошо отделены друг от друга, особенно при использовании многострочных комментариев. Стандартный текстовый режим – это 25 строк по 80 символов, поэтому для красоты программного кода и равномерного заполнения экрана желательно использовать 80 слешей (важно написать первый комментарий правильно, потому что затем, как правило, происходит копирование структуры комментария, вставка в необходимом месте и замена содержимого комментария).

 

Ещё один способ повышения удобочитаемости состоит в структуризации самого текста комментариев, это возможно при использовании текстового редактора с моноширинным шрифтом (например, UltraEdit). Выделение структуры возможно за счёт сдвига строк при помощи последовательностей пробелов. Для понимания смысла использования приведём пример комментария в заголовке программы:

 

////////////////////////////////////////////////////////////////////////////////

// …

// Алгоритм: 1. Первый шаг

//           2. Второй шаг

//              2.1. Детализация второго шага: первый шаг

//              2.2. Детализация второго шага: второй шаг

//           3. Третий шаг

////////////////////////////////////////////////////////////////////////////////

 

 В приведённом выше комментарии описывается алгоритм программы. Желательно в начале каждой программы (это касается и подпрограмм) писать её алгоритм – это даёт глубокое понимание собственной программы.

Очень эффективно писать алгоритм перед написанием программы (происходит этап проектирования). Сначала алгоритм представляет собой не подробное описание основных шагов программы. Затем по мере написания кода происходит детализация подпунктов каждого шага – это дублируется в алгоритме. Таким образом, после окончания описания программы алгоритм представляет собой развёрнутое описание программы, которое в дальнейшем поможет при необходимости расширить или исправить функционал программы, так как программисту не придётся вспоминать или разбираться с алгоритмом.

Написание алгоритма не ограничивается комментарием в заголовке программы: большие блоки операторов могут быть сопровождены детализированным алгоритмом, который является частью общего алгоритма программы.
