Запуск scp-программы

 

1.     Исполняем в рабочей папке файл make_repo.rb с ключами -f и --leave_scs (это можно сделать или при помощи командной строки, или создав командный файл *.bat со следующим содержимым: make_repo.rb -f --leave_scs). Запуск make_repo.rb формирует репозиторий, а именно создаёт папку «fs_repo_scs», являющуюся копией папки «fs_repo_src», но последняя содержит исходные коды scp-программ  в формате *.m4scp, а папка «fs_repo_scs» состоит из тех же файлов, но преобразованных в промежуточный формат *.scs. Затем создаётся папка  «fs_repo», содержащая бинарные файлы scp-программ репозитория, полученные из промежуточных файлов *.scs.

 

Рекомендация: операция формирования репозитория довольно продолжительная, так как происходит удаление всех файлов сформированного ранее репозитория, после этого репозиторий создаётся заново, обрабатывая всю библиотеку программ, включая и те программы, которые не были изменены. Выполнение этой операции при каждом запуске scp-программы приведёт к существенному увеличению времени разработки и отладки. Поэтому в большинстве случаев нужно запускать намного более быструю операцию частичного формирования репозитория (конвертируются только изменённые файлы), исполняя файл make_repo.rb без ключа –f.

Перечислим случаи, в которых следует использовать именно операцию полного формирования репозитория:

1.     Программа создана впервые.

2.     Изменился набор файлов и папок, используемых программой.

3.     Был изменён файл scp-синонимов *. scsy

4.     Внесённые в программу изменения никак не повлияли на её выполнение после операции частичного формирования репозитория.

Осторожно: если репозиторий не разу не создавался, то есть в рабочей папке отсутствуют папки «fs_repo» и «fs_repo_scs», запуск операции частичного формирования репозитория приведёт к ошибке и не создаст репозитория.

 

2.     Исполняем в рабочей папке файл scs2tgf.exe со следующими ключами:

scs2tgf -I"fs_repo_src/include" test.scs

Данный шаг производит конвертацию файла test.scs и создание нового файла test.scs.tgf, необходимого для следующего шага.

 

Рекомендация: данный шаг можно пропускать, если, во-первых, этот шаг был ранее выполнен, во-вторых, если с момента выполнения этого шага в прошлом файл text.scs не изменялся (при этом файл test.scs.tgf обязательно должен присутствовать в рабочей папке).

 

3.     Исполняем в рабочей папке файл scp.exe со следующими ключами:

scp.exe -d test.scs.tgf > _out_log

Данный шаг выполняет в виртуальной машине scp-программу, путь к которой указан в файле test.scs, и заносит все сообщения выполняемой scp-программы в файл протоколирования _out_log, который создаётся в рабочей папке и содержит системные и пользовательские сообщения scp-программы.

 

Рекомендация: все команды можно исполнять или при помощи командной строки, или создав соответствующий командный файл *.bat. Но желательно для каждой команды создавать командный файл с небольшим понятным именем, чтобы свести время введения команд к минимуму. Например, rf.bat для формирования всего репозитория (repo full), r.bat для частичного формирования репозитория (repo), t.bat создаёт test.scs.tgf (tgf), s.bat запускает интерпретатор SCP (scp). Перед отладкой необходимо запустить интерпретатор DOS-команд cmd.exe. Таким образом, каждая отладка сводится к безостановочному вводу пяти символов (‘r’=>Enter=>‘t’=>Enter=>‘s’), или даже трёх (‘r’=>Enter=>‘s’). Ввод символов можно производить даже во время формирования репозитория, так как cmd.exe имеет буфер введённых команд. После проделанных действий нужно только дождаться окончания работы scp.exe и перейти к рассмотрению лога.

Рекомендация: желательно в первый раз запустить проверенную scp-программу без ошибок (например, из репозитория) и внимательно проследить за последовательностью корректных сообщений конвертеров и scp-интерпретатора, чтобы в будущем при возникновении каких-то новых нетипичных сообщений вовремя их заметить и локализовать ошибки и предупреждения.

 

4.     После проведения всех вышеперечисленных действий результат работы scp-программы находится в файле протоколирования _out_log (далее лог).
