Содержимое исходного репозитория

 

Так сложилось, что исходные тексты, из которых собирается бинарная версия sc-памяти, хранятся на файловой системе в папке fs_repo_src. Хотя так называть папку и не обязательно, но будем отдавать дань традиции. Как вы уже знаете, на данном этапе существования технологии используется реализация sc-памяти с сегментной моделью. Есть sc-сегменты, в которых содержаться sc-элементы, и sc-директории, в которых содержатся sc-сегменты. Для удобства адресации этих трех видов объектов можно использовать URI (Uniform Resource Identifier). Например:

·         / - URI корневой директории

·         /tmp – URI системной директории, которая всегда присутствует

·         /proc – URI системной директории, которая всегда присутствует

·         /segment1 – может быть URI сегмента или директории, зависит от содержимого памяти

·         /wave_find_path_m4scp/wave_find_path – может быть URI сегмента или директории, зависит от содержимого памяти. Но так же это может быть URI sc-элемента в сегменте с URI /wave_find_path_m4scp

Как вы заметили, структура sc-памяти с этой точки зрения имеет свойство древовидности. Это позволяет хранить содержимое sc-памяти на файловой системе один в один. URI «/» соответствует корень бинарного репозитория, обычно это папка с именем fs_repo. Любой подпапке в бинарном репозитории fs_repo соответствует sc-директория с тем же именем, а любому файлу в fs_repo соответствует sc-сегменты с тем же именем (расширение является частью имени). Причем URI sc-директорий и sc-сегментов будет записываться относительно папки fs_repo, т.е. корня репозитория. Таким образом у нас есть исходный репозитория, которого получается бинарный репозиторий, на основании которого формируется содержимое sc-памяти. Рассмотрим подробно sc-репозиторий. 

Каждая папка исходного репозитория будет существовать и в бинарном репозитории, если в этой папке есть хотя бы один файл, который может быть оттранслирован в бинарную версию. Исходные файлы бывают трех типов:

·        Тип файла, при котором для формирования сегмента используются имя файла и его положение в исходном репозитории. Например, fs_repo_src/graph_theory/add_connective_visit_to_route.m4scp будет оттранслирован и будет соответствовать   fs_repo/graph_theory/add_connective_visit_to_route в бинарном репозитории. Как видите, убирается только расширение. Такие исходные тексты могут быть следующего вида (определяется по расширению):

o   *.scs – исходные текст на SCs (Semantic Code string). Линейная форма представления sc-тектов;

o   *.m4scp – исходные тексты SCP-программ, транслируемые в SCs;

o   *.scg – исходные тексты, которые помогают положить в один сегмент содержимое нескольких gwf-файлов.

·        Тип файла, для которого не формируется сегмент. Такие исходные тексты могут быть следующего вида (определяется по расширению):

o   *.scsy – заголовочные файлы синонимом, которые могут быть подключены в scs, m4scp, scg-файлы;

o   *.gwf – графический формат представления SCg-текстов.

·        Тип файла, который не подлежит обработке. Все файлы, кроме перечисленных.

 

Типы файлов scs, m4scp и scsy рассматриваются дальше, о gwf вы уже знаете, значит в данном разделе мы рассмотрим только scg-файлы. Синтаксис scg-файла очень простой. В начале файла идет блок подключения файлов синонимов с расширением scsy:

 

#include "ordinal.scsy"

#include "graph_keynodes.scsy"

 

Затем идет блок описания gwf-файлов, sc-элементы из которых должны находиться в сегменте с URI данного scg-файла в бинарном варианте:

 

testdata.gwf

test/graph.gwf

../../keynode.gwf

 

Пути записываются относительно директории, в которой лежит scg-файл. Давайте рассмотрим на примере. Предположим, что  у нас есть scg-файл fs_repo_src/seb/graph_kb.scg, в котором подключен gwf-файл fs_repo_src/seb/graph_keynodes.gwf. Таким образом, если в  graph_keynodes.gwf есть ключевой узел «маршрут*», то в sc-памяти этот ключевой узел будет иметь URI /seb/graph_kb/маршрут*.

Может возникнуть вопрос, а что если надо использовать какие-то внешние узлы, которых нету в gwf. Как тогда указать к ним путь? Это сделать очень просто. Предположим в  fs_repo_src/seb/test_graph.gwf мы используем ключевой узел с идентификатором «маршрут*», но этот ключевой узел имеет URI /graph_theory/keynodes/маршрут*. Мы пишем файл синонимом graph_keynodes.scsy и кладем его в папку «fs_repo_src/include», в которой по умолчанию ищутся файлы синонимов. В этот файл мы добавляем следующий текст:

 

“маршрут*” = “/graph_theory/keynodes/маршрут*”;

 

Только необходимо не забыть создать по пути fs_repo_src\graph_theory либо файл keynodes.scsy, либо файл keynodes.m4scp, в которых есть объявление ключевого узла. Либо можно создать scg-файл, который включает gwf-файл с данным узлом.

 

Для того, чтобы сформировать из исходного репозитория бинарную версию, достаточно выполнить в папке, где находится fs_repo_src, следующую команду:

 

builder.exe fs_repo_src

 

Если вы прописали путь к папке repobuilder в переменную PATH, то исходный репозитарий должен нормально собраться и должна появиться папка fs_repo.
